# 失败处理矩阵

> 任务编号: `T1-004`
> 状态: Draft-v1
> 最后更新: 2026-04-23

## 1. 处理原则

- 失败可分类：抓取失败、标准化失败、质量失败、发布失败。
- 失败可追溯：必须绑定 `task_id/batch_id/dataset`。
- 失败可恢复：每类失败有明确重试与人工介入策略。

## 2. 失败矩阵

| 失败类型 | 典型原因 | 系统动作 | 自动恢复 | 人工动作 |
|---|---|---|---|---|
| 抓取失败 | 源不可用/限流/超时 | 标记 `failed`，记录 source_health | 指数退避重试 | 检查源健康、调限流参数 |
| 解析失败 | schema 漂移/字段缺失 | 标记 `failed`，写入错误样本 | 可按映射版本重试 | 更新映射规则并回放 |
| 标准化失败 | 主键冲突/类型转换失败 | 标记 `partial` 或 `failed` | 可分区重跑 | 修复规则或脏数据隔离 |
| 质量失败 | 规则不通过 | 阻断发布，写 quarantine | 不自动发布 | 修复数据后重新校验 |
| 发布失败 | 写入异常/版本冲突 | 保持旧版本生效 | 重试发布事务 | 回滚并重建发布清单 |

## 3. 升级分级

- P0：服务不可用或发布版本损坏（立即回滚）
- P1：核心数据域未按 SLA 更新
- P2：局部分区质量波动

## 4. 通知与留痕

- 每次失败必须产生日志与结构化事件。
- 关键失败（P0/P1）触发告警并附 runbook 链接。
