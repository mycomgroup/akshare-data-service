# 模块迁移计划 v2

> 任务编号: `T1-007`
> 状态: Draft-v1
> 最后更新: 2026-04-23

## 1. 迁移目标

在不破坏现有 API 可用性的前提下，将旧缓存优先实现迁移到三层资产架构。

## 2. 迁移原则

- 先并行、后切换：新旧链路短期并存。
- 先核心域、后长尾域：先迁移 3 个试点数据域。
- 先文档契约、后代码重构：避免边改边漂移。

## 3. 模块映射

| 现有模块 | 目标归属 | 迁移动作 |
|---|---|---|
| `sources/` | `ingestion/` | 保留适配器，统一到执行器接口 |
| `store/` | `raw/standardized/served` | 按层拆分读写职责 |
| `offline/downloader` | `ingestion + raw` | 复用任务模型与审计写入 |
| `api.py/__init__.py` | `service/` facade | 逐步改为只读 served |
| `core/schema_registry` | `governance/` | 升级为数据契约注册 |

## 4. 分阶段计划

### 阶段 A（1-2 周）
- 完成缺失设计文档与版本模型。
- 冻结新增非试点接口。

### 阶段 B（3-6 周）
- 接入统一任务模型与 Raw 审计。
- 打通 3 个数据域标准化链路。

### 阶段 C（7-10 周）
- 启用质量门禁与 served 发布。
- 服务主路径切换到 served 读取。

### 阶段 D（11-12 周）
- 建立回放/回滚 runbook。
- 下线直连回源主路径。

## 5. 切换闸门（Go/No-Go）

- 数据一致性：新旧输出核心字段偏差在阈值内。
- 稳定性：连续 10 个交易日发布成功。
- 可观测性：关键指标与告警可用。

## 6. 回滚策略

- 服务层回滚到上一个 `release_version`。
- 任务层恢复旧调度并保留新链路日志用于复盘。
