# T0-004 变更冻结规则

> **目的**: 在重构阶段暂停新增功能，聚焦架构稳定性和代码质量，同时允许必要的 bug 修复继续进行。

---

## 1. 冻结范围

冻结期间，以下类型的新增功能**暂停开发**：

### 1.1 新增数据接口

| 类型 | 冻结内容 | 涉及文件 |
|------|----------|----------|
| **新增表定义** | 不新增 CacheTable 定义 | `src/akshare_data/core/schema.py` |
| **新增接口配置** | 不新增 interfaces/*.yaml 条目 | `config/interfaces/*.yaml` |
| **新增 API 方法** | 不新增 DataService 方法 | `src/akshare_data/api.py` |
| **新增 DataSource 方法** | 不新增 Adapter 方法 | `sources/*_source.py` |

**原因**: 当前已有 **69 个 CacheTable**、**7 个分类配置文件 (2698 行)**、**100+ API 方法**。接口数量已足够覆盖核心业务需求，新增接口会增加重构复杂度。

### 1.2 新增数据源

| 类型 | 冻结内容 | 涉及文件 |
|------|----------|----------|
| **新增 SourceAdapter** | 不新增数据源实现 | `sources/*_source.py` |
| **新增源配置** | 不新增 source 配置 | `config/sources/*.yaml` |

**原因**: 已有 4 个数据源 (Lixinger/AkShare/Tushare/Mock)，多源路由和熔断逻辑尚未完全稳定。

### 1.3 新增存储策略

| 类型 | 冻结内容 | 涉及文件 |
|------|----------|----------|
| **新增分区策略** | 不新增 partition_by 类型 | `store/strategies/*.py` |
| **新增缓存层** | 不新增存储层类型 | `store/*.py` |

**原因**: 当前 4 层架构 (memory/parquet/duckdb/incremental) 尚在优化中。

### 1.4 新增 CLI 命令

| 类型 | 冻结内容 | 涉及文件 |
|------|----------|----------|
| **新增 CLI 子命令** | 不新增 offline.cli 命令 | `offline/cli/main.py` |

**原因**: CLI 命令与底层工具耦合紧密，新增命令会影响重构测试。

## 2. 允许范围

冻结期间，以下类型的修改**允许继续**：

### 2.1 Bug 修复

| 类型 | 允许内容 | 说明 |
|------|----------|------|
| **数据获取失败** | 修复数据源返回空或异常 | 不改变接口签名 |
| **缓存读写错误** | 修复 Parquet/DuckDB 读写问题 | 不改变存储结构 |
| **字段映射错误** | 修复中英文字段映射 | 仅修改 output_mapping |
| **参数校验错误** | 修复参数类型/范围校验 | 仅修改校验逻辑 |

### 2.2 安全更新

| 类型 | 允许内容 | 说明 |
|------|----------|------|
| **依赖升级** | 升级有安全漏洞的依赖版本 | 通过 `pyproject.toml` |
| **Token 安全** | 修复 Token 泄露问题 | 仅修改 tokens.py |
| **日志敏感信息** | 移除日志中的敏感信息 | 仅修改 logging.py |

### 2.3 性能优化（不改变接口）

| 类型 | 允许内容 | 说明 |
|------|----------|------|
| **缓存命中优化** | 优化缓存查询逻辑 | 不改变 CacheManager 接口 |
| **查询性能优化** | 优化 DuckDB 查询性能 | 仅修改 duckdb.py |
| **限流策略调整** | 调整 rate_limit 参数 | 仅修改 rate_limits.yaml |

### 2.4 文档和测试

| 类型 | 允许内容 | 说明 |
|------|----------|------|
| **文档更新** | 更新 README、API 文档 | 不改变代码逻辑 |
| **测试补充** | 补充缺失的测试用例 | tests/*.py |
| **类型注解** | 补充缺失的类型注解 | 不改变逻辑 |

### 2.5 代码清理

| 类型 | 允许内容 | 说明 |
|------|----------|------|
| **删除废弃代码** | 删除未使用的函数/类 | 不影响现有功能 |
| **统一命名风格** | 统一变量/函数命名 | 通过 ruff format |
| **移除循环导入** | 修复 import 循环依赖 | 仅修改 import |

## 3. 例外流程

紧急需求可通过以下流程申请例外：

### 3.1 例外申请条件

满足以下条件之一可申请例外：

1. **生产故障**: 线上系统因缺少该接口导致核心业务中断
2. **合规要求**: 监管机构要求新增数据接口
3. **安全风险**: 缺少该接口导致安全漏洞无法修复

### 3.2 例外审批流程

```
申请人 → 提交例外申请 → 技术负责人审核 → 架构师确认 → 执行
```

**例外申请模板**:

```markdown
## 例外申请

**申请人**: [姓名]
**申请日期**: [YYYY-MM-DD]
**申请类型**: [新增接口/新增数据源/新增命令]

### 问题描述
[描述为何需要新增，不新增会导致什么问题]

### 影响范围
[列出涉及的文件和模块]

### 替代方案分析
[分析是否有不新增的替代方案]

### 例外有效期
[预计例外持续时间]

### 审批意见
- 技术负责人: [同意/拒绝/需补充]
- 架构师: [同意/拒绝/需补充]
```

### 3.3 例外执行要求

例外批准后，需满足以下要求：

1. **最小化改动**: 仅实现申请范围内的最小功能
2. **完整测试**: 补充单元测试 + 集成测试
3. **文档更新**: 同步更新 API 文档
4. **回滚预案**: 保留回滚能力

## 6. 违规处理

### 6.1 违规定义

以下行为视为违规：

1. **私自新增接口**: 未申请例外直接新增 API 方法
2. **绕过冻结检查**: 删除冻结检查代码
3. **违规合并**: 合未冻结审查的 PR

### 6.2 处理措施

| 违规级别 | 处理措施 |
|----------|----------|
| **轻微** | 回滚代码，补充例外申请 |
| **严重** | 回滚代码，暂停提交权限 1 周 |
| **重大** | 回滚代码，重新评审架构设计 |
